Anaesthesia Associates RFP · Prototype · The plan

The AA prototype: three apps, one demo

A fully interactive, true-to-life React front end over a fake in-browser backend, demonstrating every major feature of the Anaesthesia Associates Booking & Billing RFP: the scheduling canvas, BTM timesheet capture, the DRAFT→SUBMITTED→AUTHORISED approval flow, the billing engine with its three billing methods and split-billing rules, the ACCREC/ACCPAY Xero pairing, payments and disbursement, and the HL7→FHIR integration story. One app switcher in the corner flips between the Anaesthetist Mobile App (floating in a phone frame on a grey backdrop), the Anaesthetist Web App, and the Admin Web App — plus the demo-only simulators that make the invisible parts visible. Thirteen phases, each one session, each one coherent deliverable.

What we're building — and from what

The RFP (Peritia Ltd, July 2026) provides a Candidate Architecture rather than a spec: a scheduling canvas (Schedule → Day → Anaesthetist → List, two Lists per anaesthetist per day on a rolling 4-month horizon), Cards and Procedures painted onto it, a billing engine at the centre of the universe with Xero as an AR/banking bolt-on, and mobile/web/admin apps over the top. The prototype makes all of that clickable. Three documents drive every phase:

DocumentRole
docs/prototype-build/REQUIREMENTS.mdThe full requirements catalogue (P/D/M/W/A/B/X/I/N numbered) distilled from the RFP + our data-model reading. Every phase doc cites the IDs it covers.
docs/prototype-build/PROGRESS.mdThe living record: binding conventions (1–16), phase status, decisions log. Every session reads it first and updates it last — it is how each fresh chat inherits context.

Source materials for taste and truth: RFP.md (full RFP text), Data-Model-and-Flow.html (entity model, lifecycle, end-to-end flow, open questions), the RFP PDF's Appendix 3–5 screenshots (pages 44–49) showing the legacy screens, and — authoritative for all visuals — the Design/ folder at the repo root: Design Language.dc.html (tokens: palette, six status colours, type, spacing, motion) plus six interactive sample pages and a reusable iOS device frame (ios-frame.jsx). PROGRESS conventions 16–17 bind every phase to them; where a mockup simplified a business rule, the RFP rule wins (see the Decisions log).

What this prototype will NOT do

Settled up front so no phase re-opens them. The demo's honesty depends on these staying fenced.

  • No real backend. One in-memory store, seeded deterministically, persisted to localStorage, with a reset button. No servers, no fetch, no databases.
  • No real integrations. Xero, HL7/FHIR, NHI lookup, PDF parsing and OCR are all simulated with canned data and visible "demo simulation" badges. The simulators exist to demo the workflows, not the plumbing.
  • No authentication. The app switcher swaps persona (demo anaesthetist ↔ office admin). Role-based access is demonstrated through what each persona can and cannot do.
  • No pixel-cloning of the legacy screens. Ease of use is the RFP's headline requirement — screens keep the legacy information architecture (Appendices 3–5) but look modern and simple.
  • No solving the RFP's open questions. Where the RFP contradicts itself (NHI in Xero, pre-payment timing, post-op vs locked Cards…), the prototype implements the reading recorded in REQUIREMENTS §11 and labels it in UI copy as a discovery question.
  • No mobile build. The phone frame floating on a grey backdrop IS the mobile emulation — never browser resizing or devtools device mode.
  • No mega-phases. Every phase is one session with one deliverable. If a phase doc turns out to hide two deliverables, split it in PROGRESS before starting — don't push through.

How this works

  1. Run Phase 00 first — it scaffolds the app, the switcher, the phone frame and the design tokens, and puts PROGRESS.md on rails.
  2. Pick the next phase from the PROGRESS.md status table, respecting the gates: 00→01→02 strictly first; 03→04→05→06→07 in order; 08→09→10 in order; 11 any time after 07; 12 last.
  3. Open a fresh Claude Code session and paste that phase's kick-off prompt below. The agent reads PROGRESS.md, REQUIREMENTS.md and its phase doc before touching anything.
  4. The agent enters plan mode, verifies the current state of aa-prototype/ against what PROGRESS.md claims, turns the phase doc into work-sized steps, and waits for approval before changing code.
  5. Every phase ends the same way: npm run build and npx vitest run green, the phase's manual test checklist run and reported item-by-item, and PROGRESS.md updated (status row, phase entry, decisions) so the next session inherits the truth.

The checklists are the contract. Each phase doc ends with a short manual test checklist — the handful of things you personally verify in the browser before accepting the phase. The kick-off prompts require the agent to run and report that checklist, but you should click through it yourself too: it doubles as a mini demo rehearsal.
